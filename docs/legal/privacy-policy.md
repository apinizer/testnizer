# Testnizer — Privacy Policy

**Effective date:** [YYYY-MM-DD]
**Last updated:** [YYYY-MM-DD]

This Privacy Policy explains how **Testnizer** ("the Software", "we", "our")
handles information when you install and use the application. Testnizer is
published by **[Apinizer Yazılım A.Ş.]** ("the Publisher"), located in
**[İstanbul, Türkiye]**.

Testnizer is a free, open-source desktop application for testing APIs that
runs entirely on your local machine. **We do not operate any backend service
that receives, stores, or processes your data.**

---

## 1. Summary (TL;DR)

- Testnizer collects **no personal data by default**.
- All workspaces, requests, responses, environment variables, certificates,
  and credentials stay on **your local disk**.
- Outbound traffic is limited to (a) the API requests you explicitly send and
  (b) a periodic version check against GitHub Releases for auto-update.
- Telemetry / crash reporting is **disabled by default** and only activates if
  **you** provide a `SENTRY_DSN` environment variable.
- We have no user accounts, host nothing, and do not see your traffic.

---

## 2. Data We Collect

### 2.1 Data we collect from you: none, by default.

Testnizer does not transmit your collections, requests, responses, headers,
bodies, environment variables, secrets, certificates, history, or any other
content you create or import to us or any third party.

### 2.2 Data stored locally on your device

Workspaces, projects, branches, endpoints, environments, history, imported
collections (OpenAPI / Postman / Insomnia / cURL / HAR / WSDL / Proto), client
certificates and keys, application settings, and any optional master password
you enable for local data protection.

Storage locations:

- **Database (SQLite):** `<userData>/testnizer.db`
- **Settings (electron-store):** `<userData>/config.json`
- **Secrets:** OS keychain via Electron `safeStorage` (macOS Keychain,
  Windows DPAPI, libsecret on Linux)
- **Logs (if any):** `<userData>/logs/`

`<userData>` is `~/Library/Application Support/Testnizer` on macOS,
`%APPDATA%\Testnizer` on Windows, and `~/.config/Testnizer` on Linux.

You are the sole controller of this data and may delete it at any time by
removing the directory above.

### 2.3 Network traffic generated by Testnizer

Testnizer issues network requests **only** when (1) **you** explicitly send
an API request (HTTP, SOAP, WebSocket, GraphQL, gRPC, SSE) — these go
directly from your machine to the target you specify and do not pass through
us; (2) the auto-updater checks GitHub Releases (see Section 3); or
(3) optional Sentry telemetry is enabled (see Section 4).

---

## 3. Auto-Update

Testnizer uses `electron-updater` to check the project's public GitHub
Releases feed (`https://github.com/apinizer/testnizer/releases`) for newer
versions. The check sends only a standard HTTPS request: the updater's
User-Agent (app name and current version) and your IP as visible to GitHub.
**No data about your workspaces, requests, or usage is sent**, and we do not
generate any account or device identifier.

GitHub's handling of this request is governed by GitHub's own privacy policy.
You can disable the auto-updater in **Settings → Updates**.

---

## 4. Optional Telemetry (Sentry)

Testnizer ships with no default telemetry endpoint. Crash reporting only
activates when **you** (or your organization, when self-distributing) build
the application with a `SENTRY_DSN` environment variable set.

When enabled, the following may be sent to the configured Sentry instance:
application version, OS / runtime version, stack traces of unhandled errors,
and non-PII breadcrumbs (UI navigation, IPC channel names). The following are
**never** sent: request / response bodies, target URLs, headers, cookies,
tokens, certificates, environment variables, or file contents.

If you build Testnizer yourself or receive a build with telemetry enabled,
the operator of that Sentry instance is the data controller — not us.

## 5. Third-Party Services

Testnizer interacts with third parties only when triggered:

| Service | When | What is shared |
|---|---|---|
| GitHub Releases | Auto-update check | Version metadata + your IP |
| Your API targets | When you click Send | Whatever you put in the request |
| Sentry (optional) | Only if `SENTRY_DSN` is configured | Anonymous error metadata |

We have no affiliation with the API targets you choose to call.

---

## 6. Cookies and Tracking

Testnizer is a desktop application, not a website, and uses no tracking
cookies. It may store cookies returned by API targets you call (via
`tough-cookie`) so subsequent requests in the same session can reuse them;
those cookies stay on your machine.

## 7. GDPR & International Users

Because Testnizer does not transmit your personal data to us, no cross-border
transfer is initiated by the Software, and we act neither as a data controller
nor as a data processor with respect to the content you create.

The GDPR principle of **data minimization** is honored by design: the
application asks for nothing it does not strictly need. You retain full
control over your data on your device — exercising the rights of access,
rectification, and erasure by editing or deleting the local database
directly. If you enable optional telemetry through your own Sentry DSN, you
(or your organization) are the data controller for that processing.

## 8. Children's Privacy

Testnizer is intended for software developers and is not directed at children
under 13. We do not knowingly collect personal data from anyone.

## 9. Security

Local secrets are encrypted via Electron `safeStorage` (OS-level keychain).
The renderer runs with `contextIsolation: true`, `nodeIntegration: false`,
and a CSP of `connect-src 'self'`, so the UI cannot make outbound network
calls — all network activity is centralized in the main process and
originates from user-initiated actions.

No software is perfectly secure. You are responsible for the physical and
logical security of the device on which Testnizer runs.
